# Short Assignment 1: Build Challenge in Lucia

## Context
In this assignment, each team will choose **one mini-project** from the [tiny-projects](tiny-projects) folder and implement it using the **Lucia** programming language.

These projects were previously assigned in the Introduction to Programming course.

The goal of this assignment is to investigate and experiment in order to strengthen and assess your programming skills, while practicing logic, program structure, and clean design.

## Mission
Design and implement a functional version of **one** of the proposed projects, focusing on:

1. Program correctness.
2. Code clarity.
3. Proper use of Lucia language structures.
4. Console user experience (messages, flow, and validations).

## Submission rules (regardless of what is stated in the PDFs, these are the rules that apply)

1. The project must be implemented in **Lucia**.
2. It must run from a main file named `main.lucia`.
3. You may split the solution into multiple files when it improves organization.
4. Each group must choose a project that has not already been taken and reserve it by email.
5. Project assignment is based on email arrival order: the first valid email requesting a project gets that project.
6. If two groups request the same project, the second group must choose another available project.
7. Groups must be formed only by students present in class.
8. Submission deadline: 21/05/2026 no later than 06:00 p. m. (Costa Rica time), by email.
9. In addition, source code must be managed in a public GitHub repository; the repository link must be included as part of the submission.

### Required format for the project reservation email

- Subject: `TC1 - Project Reservation - [Team Name]`
- Email body:
  - Team members (full name of each person).
  - Group/section.
  - Selected project (exact PDF name).
  - GitHub repository link (if it is not ready yet, write "pending" and share it before the deadline).

## Deliverables
The submission must include at least the following:

1. `main.lucia` (entry point).
2. Supporting `.lucia` files (classes, functions, etc.).
3. A short `README.md` with:
  - Selected project name.
  - Team members.
  - How to run it.
  - Implemented features.
  - Image with the class diagram.
  - Image with the use case diagram.
  - What improvements you would make in a 2.0 version.
4. A PDF document with screenshots of software tests showing that each criterion is met.

## Evaluation criteria

| Criterion | Percentage |
|---|---:|
| Main functionality completed | 70% |
| Code quality and organization | 10% |
| Input/error handling | 10% |
| README and minimum documentation | 10% |

## Expected level
A perfect product is not expected, but the solution must be:

1. Runnable.
2. Understandable.
3. Tested by your own team.

If something is unfinished, an honest and technical explanation in the README is more valuable than hiding the issue.

## Suggestions to stand out

1. Design small functions with clear responsibilities.
2. Avoid duplicated logic (reuse functions).
3. Display clear messages to users.
4. Validate input before processing.
5. Test at least 3 different scenarios.

This assignment is designed for learning by doing: less passive theory, more building, making mistakes, iterating, and improving.

Build something you are proud to show.
