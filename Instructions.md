# Tarea Corta 1: Build Challenge en Lucia

## Contexto
En esta tarea cada equipo tomará **un mini-proyecto** de la carpeta [tiny-projects](tiny-projects) y lo implementará en el lenguaje de programación **Lucia**.

Estos proyectos fueron previamente asignados en el curso de fundamentos de programación.

La idea de esta tarea es investigar y experimentar para fortalecer y diagnosticar sus habilidades en programación, mientras practican lógica, estructura de programa y diseño limpio.

## Misión
Diseñar e implementar una versión funcional de **uno** de los proyectos propuestos, con foco en:

1. Correctitud del programa.
2. Claridad del código.
3. Buen uso de estructuras del lenguaje Lucia.
4. Experiencia de uso en consola (mensajes, flujo y validaciones).

## Reglas de la entrega (sin importar lo mencionado en los PDF, estas son las que aplican)

1. El proyecto debe estar implementado en **Lucia**.
2. Debe poder ejecutarse desde un archivo principal llamado `main.lucia`.
3. Se permite dividir en múltiples archivos cuando aporte orden.
4. Cada grupo debe elegir un proyecto que no haya sido elegido y reservarlo por correo.
5. La asignación de proyecto se realiza por orden de llegada del correo: el primer correo válido que solicite un proyecto se queda con ese proyecto.
6. Si dos grupos solicitan el mismo proyecto, el segundo grupo deberá elegir otro disponible.
7. Los grupos se forman con las personas presentes en clase únicamente.
8. Fecha límite de entrega: 21/05/2026 máximo a las 06:00 p. m. (hora de Costa Rica), por medio de correo electrónico.
9. Además, debe manejar el código fuente en un repositorio público de GitHub; como parte de la entrega debe incluir el enlace.

### Formato obligatorio del correo de reserva de proyecto

- Asunto: `TC1 - Reserva de proyecto - [Nombre del equipo]`
- Cuerpo del correo:
  - Integrantes (nombre completo de cada persona).
  - Grupo/sección.
  - Proyecto elegido (nombre exacto del PDF).
  - Enlace del repositorio de GitHub (si aún no está listo, indicar "pendiente" y luego compartirlo antes de la fecha límite).


## Entregables
El entregable debe tener el siguiente contenido mínimo:

1. `main.lucia` (punto de entrada).
2. Archivos auxiliares `.lucia` (clases, funciones, etc).
3. `README.md` breve con:
	 - Nombre del proyecto elegido.
	 - Integrantes.
	 - Cómo ejecutar.
	 - Funcionalidades implementadas.
     - Imagen con el diagrama de clases.
     - Imagen con diagrama de casos de uso.
	 - Qué mejoras harían en una versión 2.0.
4. Documento en PDF con pantallazos de las pruebas del software donde se demuestre que se cumple cada criterio.

## Criterios de evaluación

| Criterio | Porcentaje |
|---|---:|
| Funcionalidad principal completa | 70% |
| Calidad y organización del código | 10% |
| Manejo de entradas y errores | 10% |
| README y documentación mínima | 10% |

## Nivel esperado
No se espera un producto perfecto, pero sí una solución:

1. Ejecutable.
2. Entendible.
3. Probada por ustedes mismos.

Si algo no quedó listo, se valora más una explicación honesta y técnica en el README que ocultar el problema.

## Sugerencias para destacar

1. Diseñar funciones pequeñas y con responsabilidad clara.
2. Evitar repetir lógica (reutilizar funciones).
3. Mostrar mensajes claros al usuario.
4. Validar entradas antes de procesarlas.
5. Probar al menos 3 escenarios distintos.
